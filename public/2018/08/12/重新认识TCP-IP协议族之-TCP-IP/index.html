重新认识TCP/IP协议族之-TCP/IP

发表于 2018-08-12 | | 本文总阅读量 次 
作为一个码农，天天和网络打交道，有个东西其实我们无时不刻不在使用它---TCP/IP协议族。接下来我会花些时间去记录网络协议的一些底层基础知识，注意这是记录与总结底层基础，如果您自认为对这块比较熟悉，自动忽略此文章！<!-- more -->

TCP/IP最重要的子集——>http的发展历程

1.1990年划时代的http 0.9诞生，因为缺乏标准所以被定为0.9版本
2.1996年http 1.0正式发布，正真的被推崇并标准化
3.1997年http 1.1面世
2.2015年http 2.0发布
下面我来说说这几个版本的主要进步成功
step2--->step3:
    1).缓存方面：在HTTP1.0中主要使用header里的If-Modified-Since,Expires来做为缓存判断的标准，HTTP1.1则引入了更多的缓存控制策略例如Cache-Control，If-Unmodified-Since, If-Match, If-None-Match等更多可供选择的缓存头来控制缓存策略
    2).限定返回范围: HTTP 1.1允许range请求头，注意此时返回状态码的是**206**，不是200
    3).更多发送端的信息： Host，origin这些可以让接收端知道请求源在哪，虽然可以伪造，但是基本还是可以的用来，判断请求伪造以及防盗链
    4).实现长连接：不用每个请求都要牵手多次。（ps:我有一个疑问？？和websocket的长连接有区别吗？）

step3--->step4:
    1).新的二进制格式:HTTP1.x的解析是基于文本。基于文本协议的格式解析存在天然缺陷，文本格式编码多样导致处理复杂，而二进制不存在这些问题，直接使用
    2).多路复用:即连接共享，即每一个request都是是用作连接共享机制的。一个request对应一个id，这样一个连接上可以有多个request，每个连接的request可以随机的混杂在一起，接收方可以根据request的 id将request再归属到各自不同的服务端请求里面。
    3).header压缩:如上文中所言，对前面提到过HTTP1.x的header带有大量信息，而且每次都要重复发送，HTTP2.0使用encoder来减少需要传输的header大小，通讯双方各自cache一份header fields表，既避免了重复header的传输，又减小了需要传输的大小。
    4).服务端推送（server push）:HTTP2.0具有server push功能(HTTP2.0其实是https的升级版)。（ps: websocket是不是使命完成了？？）

说说http与https的进步：（此处摘自[change the world change yourself blog](https://www.cnblogs.com/zhangyfr/p/8662673.html)）
    1).https协议需要证书(付费购买)
    2).http没有ssl/tls(加密解密)层所以传输的时候是明文的
    3).header压缩 http2的类似
    4).服务端推送： http2和其类似，都是使用goggle的提议：SPADY(speedy的缩写)

http2.0较与https的改进：
    1).HTTP2.0 支持明文 HTTP 传输，而 SPDY 强制使用 HTTPS
    2).HTTP2.0 消息头的压缩算法采用 HPACK，而非 SPDY 采用的 DEFLATE,可以理解为HPACK算法算法压缩率更高，都是无损压缩

TCP/IP的通信流程（四层模型）

| TCP/IP四层概念模型 | 对应网络协议 |
| :------| ------: |
| 应用层 | HTTP、TFTP, FTP, NFS, WAIS、SMTP ,SMTP, DNS|
| 传输层 | TCP, UDP |
| 网络层 | IP, ICMP, ARP, RARP, AKP, UUCP |
| 数据链路层 | ... |

大致传输流程：
    发送端: 应用层决定用该层某协议通信，发送http请求报文 ---> 传输层把请求报文分割成多个包，并标号序号 ---> 网络层(常用IP协议)去根据要访问的ip地址与MAC地址去找到一条合适的通信路径(会有路由器的中转) --->  服务端的硬件解析
    接收端流程： 反之
